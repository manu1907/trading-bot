# Trading Bot Delivery Plan

This plan is the working roadmap. Each phase should end with a focused commit,
green tests, corrected warnings, and a short review of what changed.

## Engineering Rules

- `bot-core` is the only application module.
- Each exchange provider gets one module, starting with `bot-exchange-binance`.
- Each strategy gets one module, starting with `bot-strategy-lfa`.
- Demo and real trading use the same execution code. The difference is config,
  account, credentials, and deployment target only.
- Bot defaults, Binance capability defaults, execution/risk limits, and
  strategy defaults belong in `config/catalog.json` as much as possible.
  Environment files and runtime overrides select or narrow those defaults; they
  must not create a second implementation path.
- Local `api.env` may provide smoke-test credentials, but it must stay ignored
  and must not be used as source-controlled configuration.
- No dry-run or toy execution path should be introduced as a substitute for the
  real engine. Backtest/simulation is a separate runtime mode.
- Every phase must include pertinent unit tests and integration tests with clear
  input data and expected output.
- Before commit: run the relevant Gradle tasks, keep tests green, fix warnings
  rather than suppressing them, and avoid unused imports or dead code.
- `:bot-core:check` runs tests, Spotless, Checkstyle, and SpotBugs for core
  changes. Use affected module checks for provider/strategy changes, and use
  `./gradlew check` before broad cross-module pushes.
- After every push, check GitHub Actions for the pushed commit and wait for the
  run to finish before marking the slice complete.
- Do not hide tracked working files behind `.gitignore`. If tracked local
  notes are intentionally changed, either commit them explicitly or keep them
  out of staged slices deliberately.
- Comments should be sparse and useful: explain non-obvious decisions, not
  obvious line-by-line behavior.

## Current Seriousness Review

The current codebase has serious foundations, but it is not yet a complete
professional trading bot. The strong parts are:

- Layered config with active runtime target, override validation, schema
  versioning, and atomic runtime writes.
- Provider/strategy module boundaries.
- Binance config validation, request signing, server-time sync, exchangeInfo
  parsing, rate-limit header tracking, basic order REST calls, user-data stream
  lifecycle, and WebSocket endpoint planning.
- Schema-first Avro events, Redpanda routing, Schema Registry encoding,
  consumer/dead-letter flow, Chronicle journaling, startup recovery, restart
  recovery coverage, and deterministic archive object naming.
- CI security gates and local quality gates.

The bot is not professional-execution-ready until these gaps are closed:

- No risk-gated execution state machine yet.
- No idempotent order-command lifecycle from strategy signal through exchange
  result, user-stream execution report, reconciliation, and final position
  state.
- No account/position synchronization projection.
- No PostgreSQL/TimescaleDB projection store.
- No GCS uploader/archive worker; only the object layout exists.
- No market-data/user-data WebSocket payload mapping into Avro events.
- No LFA strategy lifecycle, feature calculations, or signal tests.
- No deterministic backtest runtime.
- No production secrets integration, metrics, alerts, deployment manifests, or
  runbooks.

Each future phase should close one of these gaps without weakening the already
established safety boundaries.

## Binance Documentation Parity Review

Source-of-truth docs checked during this review:

- Spot REST API:
  `https://developers.binance.com/docs/binance-spot-api-docs/rest-api`
- Spot user data stream:
  `https://developers.binance.com/docs/binance-spot-api-docs/user-data-stream`
- Margin new order:
  `https://developers.binance.com/docs/margin_trading/trade/Margin-Account-New-Order`
- USD-M futures new order:
  `https://developers.binance.com/docs/derivatives/usds-margined-futures/trade/rest-api/New-Order`
- COIN-M futures new order:
  `https://developers.binance.com/docs/derivatives/coin-margined-futures/trade/rest-api/New-Order`
- USD-M and COIN-M futures listen-key keepalive/close pages under
  `https://developers.binance.com/docs/derivatives/.../user-data-streams/`

Implemented Binance foundations:

- Product family modeling for Spot, Cross Margin, Isolated Margin, USD-M
  Futures, COIN-M Futures, and Options.
- HMAC SHA-256, RSA SHA-256 PKCS#8, and Ed25519 signing.
- Timestamp/recvWindow handling, server-time offset, and deterministic
  signature payload construction.
- `exchangeInfo` fetch/parse for symbol metadata, filters, rate limits, order
  types, time-in-force, trigger protection, and precision fields.
- Basic order create/query/open/cancel endpoints.
- Spot pegged-order command fields, validation, and request construction.
- Margin `sideEffectType` and `autoRepayAtCancel` command fields, validation,
  and request construction.
- Futures `workingType` and `priceProtect` command fields, validation, and
  request construction.
- Futures position-mode, margin-type, initial-leverage, and USD-M multi-assets
  mode request clients with catalog-backed paths, enums, and leverage bounds.
- Standard USD-M and COIN-M futures config rejects portfolio-margin accounts
  instead of routing `/papi` semantics through the standard futures connector.
- Spot, margin, USD-M futures, and COIN-M futures all-orders and account-trade
  history clients with catalog-backed paths for later reconciliation.
- Spot commission-rate client with catalog-backed path and typed parsing for
  standard, special, tax, and BNB-discount commission fields.
- Spot prevented-match history client with catalog-backed path, documented
  query-shape validation, and typed STP match parsing.
- Spot amend-keep-priority client with catalog-backed path, documented
  identity/new-quantity validation, and typed amended-order parsing.
- Spot cancel-replace client with catalog-backed path, reused replacement-order
  validation, documented cancel mode/restriction validation, and typed
  success/partial-failure parsing.
- Spot SOR order and SOR test-order clients with catalog-backed paths,
  SOR-specific request validation, typed SOR fill parsing, and optional
  commission-rate test parsing.
- Spot OCO order-list client with catalog-backed path, documented leg-shape
  validation, and typed order-list/order-report parsing.
- Spot OTO order-list client with catalog-backed path, documented working and
  pending order validation, and shared typed order-list/order-report parsing.
- Spot OTOCO order-list client with catalog-backed path, documented working
  order and pending OCO-pair validation, and shared typed order-list parsing.
- Spot OPO order-list client with catalog-backed path, documented working
  order and pending order validation, and typed order-list parsing.
- USD-M and COIN-M futures cancel-all and countdown-cancel-all clients with
  catalog-backed paths for outage and emergency-order cleanup workflows.
- USD-M and COIN-M futures batch-order placement clients with catalog-backed
  paths, max-5 batch validation, reused new-order validation, and per-item
  error parsing.
- USD-M and COIN-M futures modify-order clients with catalog-backed paths,
  documented order-id/client-order-id identity, LIMIT modification fields, and
  price-match collision validation.
- USD-M and COIN-M futures modify-multiple-orders clients with catalog-backed
  paths, max-5 batch validation, reused modify-order validation, and per-item
  error parsing.
- USD-M and COIN-M futures modify-history clients with catalog-backed paths,
  documented order-id/client-order-id query shape, max-100 limit validation,
  and amendment before/after parsing.
- USD-M and COIN-M futures cancel-multiple-orders clients with catalog-backed
  paths and documented order-id/client-order-id batch-shape validation.
- USD-M and COIN-M futures balance, account-information, and position-risk
  snapshot clients with catalog-backed paths for risk and reconciliation.
- USD-M and COIN-M futures ADL quantile and force-order history clients with
  catalog-backed paths for liquidation/ADL risk monitoring.
- USD-M and COIN-M futures income-history and funding-rate-history clients
  with catalog-backed paths for funding-cost and realized-PnL reconciliation.
- Catalog-backed order feature defaults for price-match, futures trigger
  controls, pegged-order, and margin side-effect controls.
- REST rate-limit header observation.
- User-data listen-key/listen-token lifecycle where configured.
- Public WebSocket stream URI planning and reconnect/rollover policy.

Binance parity gaps to close before claiming full connector support:

- Spot: implement OPOCO order lists and WebSocket API order placement.
- Margin: add borrow/repay, transfers, margin order lists, margin account/risk
  endpoints, prevented matches, and special low-latency key workflows.
- Options: verify current `/eapi` coverage against current Binance options
  documentation before enabling real options trading; keep it disabled unless
  order, MMP, account, and stream semantics are fully tested.
- All products: map user-data and market-data stream events into core Avro
  events and reconcile them against REST snapshots.
- All products: enforce exchange filters in risk/execution before order
  submission; do not rely on Binance rejections as validation.

## Target Stack

- Runtime: Java 25, Spring Boot 4, Gradle 9.5
- Provider module: Binance first, later one module per exchange
- Strategy module: LFA first, later one module per strategy
- Messaging: Redpanda using the Kafka API
- Serialization: Avro with Schema Registry
- Journal: Chronicle Queue for critical append-only event history
- Query storage: PostgreSQL/TimescaleDB or Cloud SQL PostgreSQL
- Archive storage: Google Cloud Storage
- Secrets: Google Secret Manager
- Container runtime: Docker locally, Kubernetes on Google Cloud
- Deployment: Artifact Registry, Cloud Build, GKE, Cloud Deploy or `gke-deploy`
- Observability: structured JSON logs, metrics, traces, alerts

## Phase 1: Complete Config Model

Commit:

```text

```

Deliver:

  REST endpoints, WebSocket endpoints, user-data stream lifecycle, timeouts,

Tests:

  missing provider config, disabled environment/account/market.

- Unsafe runtime reloads are rejected without replacing the active config.
