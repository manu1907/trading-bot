# Trading Bot Delivery Plan

This file is the durable handoff plan for future chats. Keep it current after every completed phase. It must tell a new chat what exists, what is deliberately blocked, and what to do next.

## Operating Rules

- Demo and real trading must use the same codebase and the same product behavior. Demo is the preproduction environment for real trading: any exchange operation, strategy behavior, remediation behavior, symbol-universe rule, risk gate, lifecycle state, and deployment path intended for real must first be enabled and validated in demo. Switching between demo and real must be done through profile, configuration, credentials, endpoints, deployment controls, Binance product availability, and calibrated risk values only.
- The bot is not a toy reporting engine. V1 online trading is the `live` profile; demo and real are runtime environments selected by config, credentials, endpoints, deployment controls, risk values, and provider product availability. Backtesting and historical simulation are deferred to v2 and are not v1 deliverables.
- Source-controlled defaults live in `config/catalog.json` and execute when no higher-precedence config source overrides them. Runtime environment config overlays the catalog and should select, narrow, or calibrate behavior without forking product logic. Effective runtime behavior is the merged result of catalog values, environment/application config, runtime target overrides, environment variables, and command-line arguments in precedence order.
- Runtime assembly belongs in `bot-app`. `bot-core` must stay provider/strategy-agnostic and must not depend on or assemble concrete provider or strategy modules.
- Provider modules declare and validate exchange capability, translate provider commands, and execute provider operations. Provider modules must not contain strategy logic.
- Strategy modules emit provider-agnostic intent or signals. Core services handle risk, reconciliation confidence, projection, planning, idempotency, and execution admission.
- Production strategy and execution must not be BTC-only. The supported trading universe must be runtime-configurable and provider-capability-aware, with eligible instruments selected by liquidity, spread, market availability, risk limits, and operational readiness. Demo must exercise that same intended real universe before real promotion; a BTCUSDT-only bootstrap is not sufficient promotion evidence.
- Exchange-side remediation must never bypass `OrderExecutionPipeline`, `OrderRiskGate`, idempotency, the event journal, projection, reconciliation confidence, or the provider gateway.
- Every phase should run focused tests where relevant, `spotlessApply`, and the full Gradle check before commit.
- After every push, verify GitHub Actions before treating the phase as complete.
- Update this file after every meaningful capability change so a new chat can resume without reading commit history.

### Events, Projection, And Recovery

- Core trading state is modeled through persisted events and replayable projections.
- Order command, risk decision, order result, reconciliation, manual intervention, manual review, remediation recommendation, and remediation decision events are projected.
- Replay rebuilds the order projection without issuing live side effects.
- Unknown exchange outcomes are represented as explicit projection markers instead of being treated as successful or failed silently.
- Projection-backed duplicate detection helps prevent repeated command effects after restart.
- Order projection now retains side and order type from command and execution-report events, and file/JDBC snapshots persist those fields so restart recovery can still build safe managed-order amendment commands.
- Unknown order results now retain the originating command action when available, and projection preserves pending `MODIFY` action metadata on unknown modify outcomes so follow-up remediation can block repeat amendments until reconciliation updates the target.
- Execution-report realized PnL attributes (`realizedProfit`, `realizedPnl`, and `realized_pnl`) are accumulated into durable account-scope and symbol-scope daily realized PnL projection state keyed by provider, environment, account, market, optional symbol, and UTC trading day.
- File and JDBC projection snapshots include daily realized PnL state; the JDBC schema resource now includes balance, position, order, risk, daily realized PnL, manual-review decision, remediation decision, pause-governance, and applied-event-id tables.
- Recovery coverage now verifies that a file projection snapshot containing an external-order intervention and a matching automated remediation decision can be restored into a fresh projection, then the scheduled remediation runner evaluates the restored decision before publishing new decisions and skips republishing a duplicate decision for the same recommendation event.
- Recovery coverage now verifies that active symbol pause governance restored from a file projection snapshot remains effective at the order risk-gate admission boundary after restart.

### Execution And Risk

- `OrderCommand` execution is routed through an execution pipeline with a risk gate before provider submission.
- The risk gate evaluates order limits, target order constraints, reconciliation confidence, manual interventions, unresolved command state, and unknown order outcomes.
- Runtime idempotency tracking exists, with projection-backed checks for restart recovery.
- Gateway failures publish unknown order result markers so reconciliation can resolve the true exchange state later.
- External-order remediation `CLOSE` can now be submitted as a cancel command through the existing execution pipeline when executor policy explicitly permits it.
- Ordinary unmanaged or externally intervened target-order commands remain blocked or manual-review according to target-order policy unless they are the narrow executor-originated remediation cancel described above.

### Binance Futures Foundations

- Binance provider foundations exist for REST signing, server time handling, exchange information, order gateway behavior, user-data handling, and reconciliation support.
- Binance USD-M futures demo usage is the current documented target profile.
- Provider capability and exchange metadata constrain Binance `NEW` order construction and validation before live exchange calls through gateway preflight and provider-side submission validation. Binance preflight also validates `CANCEL` target identity and futures `MODIFY` target/parameter requirements before gateway submission.
- Binance order execution already supports cancel commands routed by target client order id and/or exchange order id through the provider gateway.

### Manual Intervention Detection

- External or user-created orders and positions can be projected as interventions.
- Managed order changes can be detected and surfaced as intervention state.
- Operator-facing intervention state includes orders, positions, manual review items, recommendations, decisions, generated command plans, and executor reports.
- Operator acknowledgement endpoints exist for intervention records.

### Manual Review And Remediation Recommendations

- Manual review decisions can be recorded and projected.
- The remediation advisor recommends actions for external orders, managed order changes, flat positions, open positions, unknown positions, and manual review outcomes.
- Supported remediation action labels include `OPERATOR_REVIEW`, `REPLAN_FROM_PROJECTION`, `HEDGE_OR_REPLAN`, `ADOPT`, `AMEND`, `REDUCE`, `CLOSE`, `HEDGE`, `PAUSE_SYMBOL`, `PAUSE_ACCOUNT`, and `IGNORE`.
- Recommendations remain advisory until a decision is recorded.

### Automated Remediation Decisions

- A remediation decision service can publish decisions from current matching recommendations.
- An automated decision service can scan current recommendations and publish remediation decision events when explicitly enabled.
- Automated decisions dedupe by recommendation event id.
- Automated decision batch size is capped by `max_decisions_per_run`.
- `OPERATOR_REVIEW` is skipped by default for automation unless config explicitly allows publishing review decisions.
- The operator API can trigger automated decision publication with `POST /internal/interventions/remediation/automated-decisions`.
- Automated decision publication does not execute exchange operations by itself.
- A scheduled automated remediation runner now exists and is disabled by default.
- When enabled, the runner executes already-projected remediation decisions first, then publishes new automated decisions for the next projected tick.
- The runner can use the active runtime target or an explicit configured target under `trading.intervention.automated-remediation-runner.target`.
- `PAUSE_SYMBOL` and `PAUSE_ACCOUNT` remediation decisions are projected into durable pause governance state.
- Pause governance state is included in file/JDBC projection snapshots and can be restored after restart.
- Pause governance now suppresses strategy-planned order commands for paused accounts or symbols.
- The order risk gate rejects non-cancel order commands for paused accounts or symbols, while allowing valid cancel commands so the bot can reduce existing order risk.
- Active pause governance can now be released through the operator API; release publishes a remediation decision event and projects the pause state as inactive for replay/restart safety.
- Pause governance can now expire through a `pause_expires_at` ISO-8601 instant decision attribute; expired pauses remain auditable in projection but no longer block strategy or order admission.
- Pause governance override policy now exists in the order risk gate. It is disabled by default and requires explicit override actor, reason, and bounded expiry command attributes when enabled.
- Pause activation decisions, successful pause releases, explicit pause override attempts, and observed pause expiry transitions now emit structured audit log events through `AuditLogger`.
- Pause release publication outcomes and explicit pause override evaluations now emit Micrometer counters for Prometheus scraping through Actuator.
- Remediation executor preview/execute evaluations now emit `trading.remediation_executor.outcome.events` counters tagged by provider, environment, account, market, mode, operation, status, and bounded reason. ### Remediation Command Planning And Execution

- `InterventionRemediationCommandPlanner` converts persisted remediation decisions into internal command plans.
- Command plan statuses include `READY`, `NO_ACTION`, `NOT_SUPPORTED`, `STALE_PROJECTION`, and `INSUFFICIENT_DATA`.
- The planner validates that the current projection still matches the decision target before producing a ready plan.
- The planner refuses stale targets, missing intervention records, invalid position quantities, and unsupported actions.
- Order-scope `CLOSE` decisions for projected external order interventions produce `CANCEL_ORDER` plans with `exchangeExecutable=true` and `exchange_execution_path=order_execution_pipeline`.
- One-way position `CLOSE` decisions for projected external position interventions produce `CLOSE_POSITION` plans with `exchangeExecutable=true` only when the position-order policy enables one-way reduce-only execution for the matching provider/market/position side/order type. The generated plan uses the full projected absolute position amount, the opposite order side, `MARKET` order type, `reduceOnly=true`, and `closePosition=false`.
- One-way position `REDUCE` decisions for projected external position interventions produce `REDUCE_POSITION` plans with `exchangeExecutable=true` only when the position-order policy enables one-way reduce-only execution and the decision includes explicit `reduce_quantity` or `reduce_fraction` bounded by the projected absolute position amount.
- Hedge-mode position `CLOSE` and bounded `REDUCE` plans for `LONG` or `SHORT` position sides can now become exchange-executable when `position_order_policy.hedge_mode_execution_enabled=true` and projected account position-mode metadata matches `required_position_mode=HEDGE`. They use bounded projected sizing, the opposite order side, `MARKET` order type, `positionSide=LONG` or `SHORT`, `reduceOnly=false`, and `closePosition=false`, then route through `OrderExecutionPipeline`. Position `HEDGE` and `HEDGE_OR_REPLAN` can construct opposite-position-side hedge-mode market orders only when both `hedge_mode_execution_enabled=true` and `hedge_position_order_enabled=true`, and they require the same projected position-mode proof.
- Pause actions, ignore, hedge, and replan actions remain non-executable governance or planning intents. External order adoption is a gated projection ownership-transfer acknowledgement, not an exchange command. Order amendment is now policy-planned and exchange-executable for bounded managed-order interventions when `managed_order_amendment_policy` qualifies the target and projected side/order type/current price/current quantity are available.
- The operator API exposes generated plans with `GET /internal/interventions/remediation/plans`.
- `InterventionRemediationExecutorService` consumes persisted remediation decisions, regenerates current command plans, applies executor policy gates, caps batches, and returns auditable outcomes with compact plan, target, policy, and ambiguous-outcome summary attributes.
- `InterventionAutomatedRemediationRunner` can call the executor on a schedule when explicitly enabled, but it still cannot bypass executor policy, risk gates, idempotency, projection, journal, reconciliation, or provider execution.
- Executor reports can mark plans as `BLOCKED`, `PREVIEW_ONLY`, `SUBMITTED_TO_PIPELINE`, or `NO_ACTION`.
- The executor can submit `CANCEL_ORDER`, `CLOSE_POSITION`, `REDUCE_POSITION`, and `AMEND_ORDER` only when policy is enabled, exchange execution is enabled, report only is false, the exact operation is allowlisted, the target identity is present, the environment is allowed, and `OrderExecutionPipeline` is available.
- Submitted remediation cancels, position orders, and managed amendments use deterministic remediation command ids and idempotency keys derived from the remediation id.
- Managed amendment commands are submitted as `MODIFY` order commands with target client/exchange identity, projected side/order type, requested or retained price, requested or retained quantity, and remediation source attributes for risk-gate auditing.
- Managed amendment planning now blocks repeat amendment while the target has a pending `MODIFY` command or an unknown `MODIFY` result, returning explicit reconciliation blockers instead of submitting another modify command.
- If `managed_order_amendment_policy.cancel_replace_on_unsupported_change=true`, unsupported direct-amendment shapes are still blocked with `managed_order_amendment_cancel_replace_fallback_not_supported`; the bot does not silently replace orders with a multi-step cancel/new flow.
- The executor does not directly call an exchange gateway. It submits to `OrderExecutionPipeline`; the normal risk decision, provider preflight, provider gateway, event bus, journal, projection, and reconciliation path remains authoritative.
- Position remediation planning now has bounded sizing metadata for `CLOSE`, `REDUCE`, `HEDGE`, and `HEDGE_OR_REPLAN`: close uses the full projected absolute position amount, reduce requires explicit `reduce_quantity` or `reduce_fraction` bounded by the projected amount, and hedge defaults to the projected absolute amount while marking hedge-mode requirements.
- Disabled or mismatched position-order policy now blocks position close/reduce execution at planning time with explicit blockers such as `position_order_execution_policy_disabled`, `position_order_market_policy_missing`, or `hedge_mode_reduce_only_position_side_unsupported`.
- Position close/reduce plans now apply planner-side position-order risk caps before becoming exchange-executable: non-empty `allowed_symbols` must contain the projected symbol, `max_position_quantity` must not be exceeded, `max_position_notional` is estimated from projected mark price and target quantity, and `reject_unbounded_position_notional=true` blocks capped notional checks when mark price is missing or invalid. For `CLOSE` only, `chunk_close_when_max_quantity_exceeded=true` converts an oversized full close into a capped close chunk with `position_sizing_policy=bounded_projection_chunked_close`; explicit `REDUCE` decisions still block when their requested size exceeds the cap. Optional `max_account_position_notional` and `max_symbol_position_notional` caps evaluate projected gross open-position notional after the remediation action and are risk-reduction aware, so close/reduce actions that lower already-excessive exposure can proceed while additive hedge exposure is blocked when it breaches the cap. Optional `max_account_unrealized_loss` and `max_symbol_unrealized_loss` caps evaluate current projected open-position unrealized PnL and block non-reducing hedge remediation once current loss caps are breached, while still allowing risk-reducing close/reduce plans. Optional `max_account_daily_realized_loss` and `max_symbol_daily_realized_loss` evaluate projected UTC account/symbol daily realized PnL and block non-reducing hedge remediation once realized loss exceeds configured caps, while still allowing risk-reducing close/reduce plans. Optional `min_account_margin_balance` evaluates projected account margin balance and blocks non-reducing hedge remediation below the configured equity floor, while still allowing validated risk-reducing close/reduce plans. Optional `max_account_margin_drawdown_fraction` compares projected account margin balance with the stored max margin-balance high-watermark retained in risk projection snapshots and blocks non-reducing hedge remediation when drawdown exceeds the configured fraction. Position projection and JDBC snapshots now retain observed `leverage`, `margin_type`, `isolated_margin`, `position_mode`, and account max margin balance, and position-order policy can block execution when configured required margin type, required position mode, leverage bounds, projected exposure caps, current unrealized-loss caps, account/symbol daily realized-loss caps, account margin-balance floor, account margin drawdown cap, or account margin-utilization cap are missing or violated.
- The order risk gate now recognizes narrowly scoped executor-originated position close/reduce/hedge remediation commands so the external-position intervention being corrected does not block its own command. The exception requires `POSITION` scope, `CLOSE`, `REDUCE`, `HEDGE`, or `HEDGE_OR_REPLAN` action, `CLOSE_POSITION`, `REDUCE_POSITION`, or `HEDGE_POSITION` operation, `NEW MARKET`, bounded quantity, `closePosition=false`, remediation identity attributes, and either one-way `positionSide=BOTH` with `reduceOnly=true` and `position_execution_mode=one_way_reduce_only`, hedge-mode close/reduce with `positionSide=LONG/SHORT`, `reduceOnly=false`, and `position_execution_mode=hedge_mode_position_side_close_reduce`, or hedge-mode hedge with `position_execution_mode=hedge_mode_position_side_hedge`.
- The executor still does not directly mutate projection ownership, acknowledge interventions, perform cancel/replace amendment fallback, pause symbols/accounts, or execute any real-environment command unless policy explicitly allows real. External order adoption is handled by the remediation orchestrator through an auditable acknowledgement when `order_adoption_acknowledgement_enabled=true`.

## Current Limitations And Explicit Blocks

- The bot is not yet ready for unattended real-money autonomous trading.
- External-order `CLOSE`, one-way position `CLOSE`, bounded one-way position `REDUCE`, bounded managed-order `AMEND`, config-gated hedge-mode position `CLOSE`, config-gated bounded hedge-mode position `REDUCE`, and separately config-gated position `HEDGE` remediation can currently become executable exchange commands, and only through `OrderExecutionPipeline`.
- Automated remediation can now run on a schedule when explicitly enabled, but exchange execution still requires the executor execution path and explicit executor policy.
- Order adoption is not exchange-executable; it is a replayable projection ownership transfer for matching external orders. Order amendment controls have a catalog-disabled planner policy and a bounded executor path; the checked-in bootstrap demo runtime currently enables strict BTCUSDT/ETHUSDT managed-limit-order amendments plus adopted-order cancel/amend lifecycle controls for BTCUSDT/ETHUSDT while keeping adopted replace disabled. Adopted pending/unknown modify outcomes now carry explicit reconciliation and rollback-disposition metadata, but exchange-executable adopted replace/rollback remains unsupported unless a separate safe design is accepted. This initial two-symbol universe must be replaced or expanded to the same configurable high-liquidity symbol universe intended for real before demo can be used as real-promotion evidence.
- Scheduled live remediation is currently limited to projected decisions and the currently supported external-order cancel, one-way reduce-only position close/reduce, bounded managed-order amendment, explicitly enabled hedge-mode position-side close/reduce, and separately enabled hedge-order execution paths.
- Pause actions create durable pause governance state, enforce admission blocks, support operator release, support time-bounded expiry attributes, disabled-by-default override policy, structured activation/release/override/expiry audit logs, release/override/activation/expiry-configured/expiry-transition/audit-store-failure counters, active-pause gauges, a recent audit-query endpoint, optional JSONL-backed audit persistence, optional indexed JDBC audit persistence, Prometheus alert rules, Alertmanager routing through PagerDuty/Slack/email, Google Cloud and AWS Secret Manager binding names for demo alert receiver substitutions, Google Cloud Secret Manager binding names for real alert receiver substitutions, a Google Cloud Alertmanager renderer that validates placeholders and renders demo/real configs from Secret Manager without printing secret values, and an importable Grafana dashboard. Adopted orders are identified as managed targets without a bot command id, and ordinary target-order commands are blocked by default unless `target_order.allow_adopted_target_orders=true`. - Manual/external one-way positions can be evaluated and automatically closed or reduced through bounded reduce-only market orders when demo executor policy allowlists the operation; manual/external hedge-mode positions can be evaluated and automatically closed or reduced through bounded position-side market orders when a runtime explicitly enables hedge-mode execution.
- The production operations package is not complete yet: remaining work includes
  real alert receiver values, broader metrics/health checks beyond the first
  runtime readiness endpoint/metrics/dashboard, strategy/account dashboards,
  live incident drill evidence, and reproducible infrastructure-as-code beyond

- The LFA exchange-polled universe admission, portfolio selection, full money management, aggregate risk budgeting, and position-management loop are not complete enough to treat it as a production autonomous strategy. A first conservative projected top-of-book imbalance analyzer, config-gated live signal runner, validated lifecycle gate with explicit non-entry blockers, operator-token-protected lifecycle inspection/transition endpoints with durable `STRATEGY_LIFECYCLE` event projection, projected-data warm-up gate, exchange-polled and market-data-liquidity-gated instrument admission, first metadata-derived provider stream coverage with scheduled derived-stream rebalancing, first projected account-margin market-quality allocation layer, aggregate strategy-run notional cap, runner-level reconciliation confidence gating, first account/symbol budget gates, first projected risk/money-management fit ranking, first top-of-book expected-profit ranking, and configurable expected-profit and risk-fit admission floors plus stale-signal expiry exist, but the checked-in demo runtime keeps the runner disabled until full position management and broader money-management controls are implemented.
