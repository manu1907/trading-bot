# Trading Bot Delivery Plan

This file is the durable handoff plan for future chats. Keep it current after every completed phase. It must tell a new chat what exists, what is deliberately blocked, and what to do next.

## Operating Rules

- Demo and real trading must use the same codebase. Switching between them must be done through profile, configuration, credentials, and deployment controls only.
- The bot is not a toy dry-run engine. Simulation and backtesting are separate runtimes; live/demo trading must exercise the same execution, risk, journal, projection, and reconciliation paths.
- Source-controlled defaults live in `config/catalog.json`. Runtime environment config should select and narrow behavior, not fork product logic.
- Provider modules declare and validate exchange capability, translate provider commands, and execute provider operations. Provider modules must not contain strategy logic.
- Strategy modules emit provider-agnostic intent or signals. Core services handle risk, reconciliation confidence, projection, planning, idempotency, and execution admission.
- Exchange-side remediation must never bypass `OrderExecutionPipeline`, `OrderRiskGate`, idempotency, the event journal, projection, reconciliation confidence, or the provider gateway.
- Every phase should run focused tests where relevant, `spotlessApply`, and the full Gradle check before commit.
- After every push, verify GitHub Actions before treating the phase as complete.
- Update this file after every meaningful capability change so a new chat can resume without reading commit history.

## Current Implemented Capabilities

### Profile And Configuration

- The runtime profile/config system supports the same application code for demo and real environments.
- Configuration catalogs include exchange/provider, execution, risk, idempotency, reconciliation, and intervention/remediation defaults.
- Operator API access can be protected with `X-Operator-Token` when enabled by config.
- Automated remediation policy is configurable under `trading.intervention.automated-policy`.
- Remediation executor policy is configurable under `trading.intervention.remediation-executor-policy`.
- Executor policy defaults keep the executor disabled, exchange execution disabled, dry-run only, real environments blocked, and executable operations empty.
- Executor policy validation requires explicit allowed operations and strict ready-plan, fresh-projection, target-identity, and managed-pipeline gates before exchange execution can be enabled.
- Target-order risk policy now includes `allow_external_remediation_cancel`, which defaults to true but only applies to executor-originated `ORDER`/`CLOSE`/`CANCEL_ORDER` commands with explicit remediation attributes.
- Conservative remediation defaults are configured: unknown or high-risk intervention types route to operator review or non-executable planning instead of live exchange action.

### Events, Projection, And Recovery

- Core trading state is modeled through persisted events and replayable projections.
- Order command, risk decision, order result, reconciliation, manual intervention, manual review, remediation recommendation, and remediation decision events are projected.
- Replay rebuilds the order projection without issuing live side effects.
- Unknown exchange outcomes are represented as explicit projection markers instead of being treated as successful or failed silently.
- Projection-backed duplicate detection helps prevent repeated command effects after restart.

### Execution And Risk

- `OrderCommand` execution is routed through an execution pipeline with a risk gate before provider submission.
- The risk gate evaluates order limits, target order constraints, reconciliation confidence, manual interventions, unresolved command state, and unknown order outcomes.
- Runtime idempotency tracking exists, with projection-backed checks for restart recovery.
- Gateway failures publish unknown order result markers so reconciliation can resolve the true exchange state later.
- External-order remediation `CLOSE` can now be submitted as a cancel command through the existing execution pipeline when executor policy explicitly permits it.
- Ordinary unmanaged or externally intervened target-order commands remain blocked or manual-review according to target-order policy unless they are the narrow executor-originated remediation cancel described above.

### Binance Futures Foundations

- Binance provider foundations exist for REST signing, server time handling, exchange information, order gateway behavior, user-data handling, and reconciliation support.
- Binance USD-M futures demo usage is the current documented target profile.
- Provider capability and exchange metadata are intended to constrain order construction and validation before any live exchange call.
- Binance order execution already supports cancel commands routed by target client order id and/or exchange order id through the provider gateway.

### Manual Intervention Detection

- External or user-created orders and positions can be projected as interventions.
- Managed order changes can be detected and surfaced as intervention state.
- Operator-facing intervention state includes orders, positions, manual review items, recommendations, decisions, generated command plans, and executor reports.
- Operator acknowledgement endpoints exist for intervention records.

### Manual Review And Remediation Recommendations

- Manual review decisions can be recorded and projected.
- The remediation advisor recommends actions for external orders, managed order changes, flat positions, open positions, unknown positions, and manual review outcomes.
- Supported remediation action labels include `OPERATOR_REVIEW`, `REPLAN_FROM_PROJECTION`, `HEDGE_OR_REPLAN`, `ADOPT`, `AMEND`, `REDUCE`, `CLOSE`, `HEDGE`, `PAUSE_SYMBOL`, `PAUSE_ACCOUNT`, and `IGNORE`.
- Recommendations remain advisory until a decision is recorded.

### Automated Remediation Decisions

- A remediation decision service can publish decisions from current matching recommendations.
- An automated decision service can scan current recommendations and publish remediation decision events when explicitly enabled.
- Automated decisions dedupe by recommendation event id.
- Automated decision batch size is capped by `max_decisions_per_run`.
- `OPERATOR_REVIEW` is skipped by default for automation unless config explicitly allows publishing review decisions.
- The operator API can trigger automated decision publication with `POST /internal/interventions/remediation/automated-decisions`.
- Automated decision publication does not execute exchange operations by itself.
- `PAUSE_SYMBOL` and `PAUSE_ACCOUNT` remediation decisions are projected into durable pause governance state.
- Pause governance state is included in file/JDBC projection snapshots and can be restored after restart.
- Pause governance now suppresses strategy-planned order commands for paused accounts or symbols.
- The order risk gate rejects non-cancel order commands for paused accounts or symbols, while allowing valid cancel commands so the bot can reduce existing order risk.
- Active pause governance can now be released through the operator API; release publishes a remediation decision event and projects the pause state as inactive for replay/restart safety.
- Pause governance can now expire through a `pause_expires_at` ISO-8601 instant decision attribute; expired pauses remain auditable in projection but no longer block strategy or order admission.
- Pause governance override policy now exists in the order risk gate. It is disabled by default and requires explicit override actor, reason, and bounded expiry command attributes when enabled.
- Pause activation decisions, successful pause releases, explicit pause override attempts, and observed pause expiry transitions now emit structured audit log events through `AuditLogger`.
- Pause release publication outcomes and explicit pause override evaluations now emit Micrometer counters for Prometheus scraping through Actuator.
- Effective active account/symbol pause counts now emit low-cardinality Micrometer gauges through `trading.pause_governance.active.states`.
- Live-only pause activation decisions now emit `trading.pause_governance.activation.events`; pause activations with a valid `pause_expires_at` now emit `trading.pause_governance.expiry.configured.events`.
- Projected active pauses crossing a valid `pause_expires_at` are now observed by a scheduled pause expiry monitor, which emits one `pause_governance_expired` audit record and one `trading.pause_governance.expiry.transitions` metric per projected pause expiry without mutating projection state or touching the exchange.
- Recent pause activation, pause release, explicit pause override, and pause expiry audit records can now be queried through `GET /internal/interventions/pauses/audit-events`.
- Pause governance audit records can now be persisted to an optional append-only JSONL file store with `trading.audit.pause-governance.file-store.enabled=true`; when configured, the audit endpoint reads persisted audit records so recent pause governance audit history remains queryable after restart.
- Audit-store persistence or query failures now emit `trading.pause_governance.audit_store.failures` tagged by `operation` and `store`, giving dashboards and alerts an explicit degradation signal.
- Prometheus-compatible pause governance alert rules now exist in `ops/prometheus/pause-governance-alerts.yml` for active pauses, activation, release publish failures, overrides, expiry transitions, and audit-store failures.

### Remediation Command Planning And Execution

- `InterventionRemediationCommandPlanner` converts persisted remediation decisions into internal command plans.
- Command plan statuses include `READY`, `NO_ACTION`, `NOT_SUPPORTED`, `STALE_PROJECTION`, and `INSUFFICIENT_DATA`.
- The planner validates that the current projection still matches the decision target before producing a ready plan.
- The planner refuses stale targets, missing intervention records, invalid position quantities, and unsupported actions.
- Order-scope `CLOSE` decisions for projected external order interventions produce `CANCEL_ORDER` plans with `exchangeExecutable=true` and `exchange_execution_path=order_execution_pipeline`.
- Position actions, pause actions, adoption, amendment, ignore, and replan actions remain non-executable governance or planning intents.
- The operator API exposes generated plans with `GET /internal/interventions/remediation/plans`.
- `InterventionRemediationExecutorService` consumes persisted remediation decisions, regenerates current command plans, applies executor policy gates, caps batches, and returns auditable outcomes.
- Executor reports can mark plans as `BLOCKED`, `DRY_RUN`, `SUBMITTED_TO_PIPELINE`, or `NO_ACTION`.
- The executor can submit only `CANCEL_ORDER` for order `CLOSE` remediation, and only when policy is enabled, exchange execution is enabled, dry-run only is false, `CANCEL_ORDER` is allowlisted, the target identity is present, the environment is allowed, and `OrderExecutionPipeline` is available.
- Submitted remediation cancels use deterministic remediation command ids and idempotency keys derived from the remediation id.
- The executor does not directly call an exchange gateway. It submits to `OrderExecutionPipeline`; the normal risk decision, provider gateway, event bus, journal, projection, and reconciliation path remains authoritative.
- The executor still does not mutate positions, acknowledge interventions, adopt orders, amend orders, pause symbols/accounts, or execute any real-environment command unless policy explicitly allows real.

### Operator API Surface

- `GET /internal/interventions/orders` lists projected order interventions.
- `GET /internal/interventions/positions` lists projected position interventions.
- `GET /internal/interventions/manual-reviews` lists manual review state.
- `GET /internal/interventions/pauses` lists projected pause governance state.
- `GET /internal/interventions/pauses/audit-events` lists recent pause governance audit records for activation, release, override, and expiry events from the optional file store when enabled, otherwise from the bounded in-memory buffer.
- `GET /internal/interventions/remediation` lists remediation recommendations.
- `GET /internal/interventions/remediation/decisions` lists remediation decisions.
- `GET /internal/interventions/remediation/plans` lists generated command plans.
- `GET /internal/interventions/remediation/executor/dry-run` lists executor outcomes without submitting commands.
- `POST /internal/interventions/pauses/releases` releases a matching active account or symbol pause through an auditable remediation decision event.
- `POST /internal/interventions/remediation/decisions` records a remediation decision.
- `POST /internal/interventions/remediation/automated-decisions` triggers automated decision publication when enabled.
- `POST /internal/interventions/remediation/executor/execute` evaluates current remediation decisions and submits eligible order-cancel remediation commands only when executor policy permits exchange execution.
- `POST /internal/interventions/orders/acknowledgements` acknowledges order interventions.
- `POST /internal/interventions/positions/acknowledgements` acknowledges position interventions.

### Documentation

- Current-state documentation exists for the bot's architecture, supported scenarios, and limitations.
- A user manual exists for starting and using the bot in the current Binance USD-M futures demo profile.
- Documentation describes that the bot can detect, recommend, decide, plan remediation, dry-run executor outcomes, and submit the first policy-gated external-order close cancel through the normal execution pipeline.

## Current Limitations And Explicit Blocks

- The bot is not yet ready for unattended real-money autonomous trading.
- Only external-order `CLOSE` remediation can currently become an executable exchange command, and only as a cancel routed through `OrderExecutionPipeline`.
- Automated remediation currently publishes decisions only; exchange execution still requires the executor execution path and explicit executor policy. There is no scheduled fully automatic executor loop yet.
- Position remediation, order adoption, and order amendment controls are not exchange-executable yet.
- Automated decision execution is currently triggered through the operator API; there is no scheduled live remediation loop yet.
- Pause actions create durable pause governance state, enforce admission blocks, support operator release, support time-bounded expiry attributes, disabled-by-default override policy, structured activation/release/override/expiry audit logs, release/override/activation/expiry-configured/expiry-transition/audit-store-failure counters, active-pause gauges, a recent audit-query endpoint, optional JSONL-backed audit persistence, and Prometheus alert rules, but they do not yet have dashboards, deployment-specific external alert routing, or a production-grade indexed audit backend.
- Position remediation does not yet have a complete bounded sizing policy for close, reduce, or hedge actions.
- Position remediation does not yet construct reduce-only orders or hedge orders.
- External order adoption semantics are not yet defined enough for live execution.
- Managed order amendment semantics are not yet defined enough for live execution.
- Manual/external positions can be evaluated, but automatic action against them remains blocked until sizing rules and executable position command policies are implemented.
- There is no complete production deployment package with secrets management, metrics, alerts, dashboards, or runbooks.
- There is no deterministic backtest runtime that proves strategy behavior across historical data.
- The LFA strategy lifecycle, feature calculation, signal generation, and position-management loop are not complete enough to treat it as a production autonomous strategy.

## Current Demo USD-M Futures User Flow

- Start the bot with the demo profile and Binance USD-M futures demo credentials.
- Use the same application code that would be used for real trading; only profile, credentials, endpoint, and risk/config values should differ.
- Inspect interventions through the operator API.
- Inspect remediation recommendations for external orders, changed managed orders, and positions.
- Record manual remediation decisions through the operator API if desired.
- Enable automated decision publication only when the configured policy is understood.
- Trigger automated decision publication through the operator API when enabled.
- Inspect generated remediation command plans.
- Inspect remediation executor dry-run outcomes.
- Execute eligible remediation batches only after enabling the executor policy, exchange execution, `dry_run_only=false`, and the relevant operation allowlist. Today the only eligible exchange operation is `CANCEL_ORDER` for an external order `CLOSE` decision.
- Treat position, pause, adoption, amendment, ignore, and replan plans as non-executing until their specific execution policies are implemented.

## End-To-End Delivery Roadmap To Live Google Cloud Operation

This section describes the complete remaining work from the current codebase to a bot that can be built by CI, deployed by CI/CD, operated on Google Cloud, run continuously against Binance USD-M futures demo, and later promoted to real trading with the same codebase. The estimate is intentionally conservative because production trading work needs verification slices, not only feature slices.

Estimated remaining delivery: 40 to 65 commits/slices.

- Core remediation execution and governance: 1 to 5 slices.
- Strategy runtime and trading lifecycle: 10 to 16 slices.
- Persistence, recovery, and replay hardening: 6 to 10 slices.
- Backtesting, paper/demo validation, and promotion gates: 6 to 10 slices.
- Observability, alerts, and operator controls: 4 to 7 slices.
- Google Cloud infrastructure and CI/CD: 8 to 12 slices.
- Security, secrets, operations, and live runbooks: 5 to 8 slices.
- Final demo-live burn-in and real-live readiness: 4 to 7 slices.

### Roadmap Slice 1: Complete Automated Remediation Governance

- Durable pause governance state exists for `PAUSE_SYMBOL` and `PAUSE_ACCOUNT`; it is enforced in strategy admission and order execution admission for non-cancel commands.
- Pause governance can be released by operator API through replayable remediation decision events.
- Pause governance can expire from replayed decision attributes.
- Pause governance override policy exists in order risk-gate admission and remains disabled by default.
- Structured audit logs exist for pause activation, pause release, explicit override attempts, and observed expiry transitions.
- Micrometer counters exist for pause release publication outcomes and explicit override evaluations.
- Low-cardinality active-pause gauges exist for account and symbol pause scopes.
- Live-only counters exist for pause activation decisions and pause activations with valid configured expiries.
- Scheduled pause expiry monitoring now emits one expiry audit record and one expiry-transition metric per projected active pause that crosses a valid `pause_expires_at`.
- Recent pause activation, pause release, explicit pause override, and pause expiry audit records are queryable from the operator API.
- Optional append-only JSONL persistence exists for pause governance audit records, and the operator audit endpoint reads persisted audit records when the store is enabled.
- Audit-store persistence and query failures emit a Micrometer counter for operational alerting.
- Prometheus-compatible alert rules exist for active pauses, activation, release publish failures, overrides, expiry transitions, and audit-store failures.
- The remaining pause governance work is dashboards, deployment-specific external alert routing, and a production-grade indexed audit backend if JSONL is not sufficient for deployment.
- Add operational dashboards and deployment-specific external alert routes for pause activation, expiry, override, release, and audit-store failures.
- Add bounded position sizing policy for `REDUCE`, `CLOSE`, and `HEDGE`.
- Implement position close/reduce/hedge planning only after sizing, reduce-only rules, margin mode, leverage, symbol capability, and account risk limits are explicit.
- Implement order adoption policy only after ownership transfer, client order id handling, reconciliation, and rollback behavior are defined.
- Implement managed order amendment policy only after allowed field drift, stale-state handling, and cancel/replace behavior are defined.
- Add scheduled remediation runner after executable remediation idempotency and pause governance are covered by persistence tests.
- Definition of done: unattended demo remediation can handle external orders and known external-position scenarios without manual review, while unsupported scenarios are automatically blocked or paused with auditable reasons.

### Roadmap Slice 2: Complete Strategy And Trading Runtime

- Finish the LFA strategy lifecycle from market data to signal generation to order intent.
- Add strategy-level risk budgets, symbol budgets, position budgets, daily loss limits, max drawdown limits, max open orders, and max exposure.
- Add lifecycle state for strategy start, warm-up, active trading, paused, draining, stopped, and emergency stop.
- Add position-management logic for entry, exit, stop, take-profit, timeout, stale signal, partial fill, and unknown result handling.
- Add strategy admission checks that read current projection, reconciliation confidence, pause governance, and risk budget state before emitting commands.
- Add deterministic strategy command ids and idempotency keys.
- Definition of done: the bot can run the strategy in demo using live data, make bounded decisions, and recover from restart without duplicate orders.

### Roadmap Slice 3: Harden Persistence, Replay, And Recovery

- Ensure all live side-effecting components are disabled during replay.
- Persist and replay remediation decisions, executor submissions, pause state, strategy lifecycle state, and risk budget state.
- Add recovery tests for restart during pending order command, gateway failure, unknown order result, partial fill, external intervention, and remediation execution.
- Add projection reconciliation after restart before any strategy or remediation executor can resume.
- Add journal retention, compaction, backup, and restore procedures.
- Definition of done: a process restart or deployment rollout does not duplicate exchange actions and does not resume trading until projection and exchange state are reconciled.

### Roadmap Slice 4: Backtesting, Simulation, And Demo Validation

- Build a deterministic backtest runtime that uses historical market data and the same strategy/risk logic, but no live exchange side effects.
- Add simulation for fills, fees, slippage, funding, partial fills, rejects, cancel races, and network faults.
- Add paper/demo validation reports for win rate, drawdown, exposure, order count, fill quality, latency, and failure handling.
- Define promotion gates from backtest to demo, from demo shadow mode to demo live trading, and from demo live trading to real trading.
- Definition of done: strategy promotion is based on repeatable evidence and configured gates, not manual confidence.

### Roadmap Slice 5: Observability And Operator Controls

- Add structured metrics for signals, commands, risk decisions, gateway calls, order results, reconciliation, interventions, remediation executor outcomes, and strategy PnL.
- Add health checks for exchange connectivity, user-data stream freshness, market-data freshness, reconciliation confidence, journal writes, and event consumer lag.
- Add alerting for emergency stop, degraded reconciliation, unknown order state, unexpected external intervention, high loss, excessive order rate, stale streams, failed deployments, and secret/config errors.
- Add operator endpoints for read-only status, pause/resume, emergency stop, controlled drain, and safe config inspection.
- Add dashboards for strategy state, account state, exposure, order lifecycle, remediation lifecycle, and deployment health.
- Definition of done: the bot can be operated without reading logs as the primary control surface.

### Roadmap Slice 6: Google Cloud Infrastructure

- Choose and codify the target Google Cloud runtime. The likely first target is Cloud Run for the application plus managed supporting services, unless continuous websocket/session behavior or latency requirements force GKE or a VM-based deployment.
- Provision environments for demo and real with separate projects or strict environment separation.
- Add infrastructure as code for service accounts, IAM, Artifact Registry, Secret Manager, Cloud Run or GKE, VPC/network controls if needed, logging, monitoring, alert policies, and deployment variables.
- Store Binance credentials and operator tokens in Secret Manager, never in source-controlled config.
- Add runtime config overlays for demo and real that select endpoints, credentials, risk budgets, executor policy, and promotion gates without changing application code.
- Define rollout, rollback, emergency stop, and disaster recovery procedures.
- Definition of done: a clean Google Cloud environment can be created reproducibly and can run the bot with secrets and config injected securely.

### Roadmap Slice 7: CI/CD Pipeline

- Keep GitHub Actions as the source CI gate for formatting, tests, static analysis, dependency checks, secret scanning, and container build checks.
- Add container image build and publish to Artifact Registry after CI passes.
- Add signed or traceable build metadata so a deployment can be tied back to a commit.
- Add deployment workflows for demo and real with separate approvals, environment variables, secrets, and rollout policies.
- Add smoke tests after deployment: app health, config load, exchange server time, market-data connectivity, user-data stream connectivity, and no unintended exchange action on startup.
- Add deployment rollback workflow.
- Definition of done: merging to the deployment branch can build, publish, deploy to demo, run smoke tests, and require explicit approval before any real deployment.

### Roadmap Slice 8: Demo-Live Burn-In

- Run the bot in Google Cloud against Binance USD-M futures demo with strategy execution disabled first, then strategy shadow mode, then bounded small-order demo trading.
- Validate that market-data streams, user-data streams, order lifecycle, reconciliation, journal replay, remediation, alerts, and dashboards work continuously.
- Track burn-in metrics for uptime, latency, reconciliation confidence, unknown order count, duplicate prevention, intervention count, remediation outcome, and strategy PnL.
- Add incident drills: restart during open order, network outage, stale user-data stream, external manual order, bad config, failed deployment, and emergency stop.
- Definition of done: demo-live operation has passed the agreed burn-in period with no unresolved critical incidents and with documented evidence.

### Roadmap Slice 9: Real Trading Readiness And Promotion

- Keep real trading disabled until demo-live promotion gates pass.
- Require explicit real environment config: real credentials, real endpoint, real risk budgets, real executor policy, real alert routing, real emergency stop, and real deployment approval.
- Start real trading with smallest practical risk limits, restricted symbol allowlist, conservative order limits, and remediation executor policy still constrained to proven operations.
- Monitor real trading with tighter alerts and a defined manual kill switch.
- Increase limits only by explicit config changes after evidence review.
- Definition of done: real trading is a controlled production rollout using the same codebase and deployment path as demo, with separate credentials, config, approvals, and monitoring.

### Roadmap Slice 10: Production Operations Maturity

- Add runbooks for deployment, rollback, emergency stop, exchange outage, reconciliation degradation, external intervention, unknown order, credential rotation, secret compromise, and incident review.
- Add periodic credential rotation and least-privilege IAM review.
- Add cost monitoring and budget alerts for Google Cloud.
- Add release notes and audit trail for every promoted deployment.
- Add long-term data retention policy for journals, logs, metrics, and backtest/demo evidence.
- Definition of done: the system is maintainable as a production service, not only runnable as a developer project.
